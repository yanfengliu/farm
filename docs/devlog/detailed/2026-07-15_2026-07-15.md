# 2026-07-15

## Restored the authoritative test gate

- Result: `npm test` was red on a clean checkout of `main` and had been reporting failures that no code change caused. On this 32-core machine the suite finished 14 files failed / 380 of 394 tests passing at commit `a0b6a6b`, with every failure an identical `Test timed out in 15000ms` rather than an assertion. The repo's first and most authoritative gate could not be trusted to distinguish a real regression from runner noise.
- Diagnosis: 16 suites under `tests/browser` each launch their own Chromium and their own Vite dev server, and the browser files set their own 15s per-test budgets inline (`}, 15000)`). The repo had no Vitest configuration of any kind, so Vitest defaulted to one fork per core and started ~31 of those suites at once. The tests were not wrong and the timeouts were not flakes in the tests themselves: the runner was oversubscribing the machine until browser work missed the budgets the tests declared for themselves.
- Fix: `vitest.config.ts` now caps `poolOptions.forks.maxForks` at `Math.max(2, Math.min(6, os.cpus().length - 1))`, so fork concurrency is bounded by the browser cost of the suite rather than by core count. Nothing else changed; no test timeout was raised and no test was skipped or weakened.
- Verification: the same suite at the same commit goes from 14 files failed to 46 of 46 files passing with the cap in place, and repeated full runs reproduce the green result rather than passing once by luck. Wall-clock cost is roughly 45-49s against ~40s for the previously-failing unbounded run, so the cap trades no meaningful speed for a trustworthy gate. `npm run typecheck` and `npm run lint` are clean and `npm run build` retains only the established Phaser chunk-size advisory.
- Regression anchor: `tests/browser/testRunnerConcurrency.test.mjs` pins that the fork cap stays a small bounded number and stays below the number of suites that launch their own Chromium, so the cap cannot silently regrow back to core count. A mutation that restores the uncapped `os.cpus().length - 1` fails the anchor with `expected 31 to be less than or equal to 8` and `expected 31 to be less than 17`; restoring the cap makes it green again.

## Closed out the visual-loop screenshot guard

- Inherited state: an uncommitted guard detected "compositor-corrupt" screenshots and retried them, failing closed by throwing. Two independent adversarial reviewers read it against live code and the empirical record, and both refused it. Its intent was sound and its cost was not the problem — a reviewer benchmarked the in-page pixel scan at ~14ms median against a real 1280x800 frame, one dropped frame per capture and ~2.3s across a 161-frame run, so the decode stays in the page rather than pulling in a PNG dependency and its audit surface.
- Detection was the real defect: the run-length trips were ANDed behind a `>= 0.005` coverage floor, so a band of pure black spanning the entire frame width (`longestBlackRowRatio` 1.0) and a full-height black column were both reported clean, as was a 400x10 strip whose row ratio of 0.3125 already satisfied the geometric condition. Only a 256x256 tile tripped it. The guard missed the exact corruption class it was named for. The floor is also unjustifiable against measured baseline: all 46 frames of the recorded 2026-07-14 run report `exactBlackPixels` of exactly 0, the Phaser clear color is `#3f5f32`, and every black in the stylesheet is a translucent shadow, so 0.005 sat 5,120 pixels above a baseline of nothing. Coverage and run-length trips are now independent at 0.002 and 0.25, and all four previously-missed frames flag while a real frame stays clean.
- Blast radius was the second defect: `run.finalObservation` is captured inside a `try` whose only handler is `finally`, so throwing there skipped the bundle export, replay self-check, manifest, all three report writes, and the ledger append. One unstable frame at the final capture discarded a completed 160-decision run's entire report. The in-loop call site was already safe because civ-engine's runner converts a host throw into a `hostError` stop. Capture now keeps the least-corrupt attempt, marks it `degraded`, and `evaluateVisualLoop` raises a medium `screenshot-compositor-degraded` finding; fail-soft without that finding would be strictly worse than the old behavior, because it would archive a corrupt frame in silence.
- Synchronization was the third defect: retrying only the screenshot advanced the world by ~83ms per settle between the DOM sample and the archived frame, contradicting the packet guarantee the changelog and testing guide both state. Retrying the whole observe-and-shoot attempt restores it, and the existing contract test `visual loop samples DOM observation on a rendered frame before writing the screenshot` now passes unmodified because the ordering is genuinely restored rather than relaxed to fit.
- Signal was the fourth defect: `screenshotCapture` was written on every observation and read by nothing, adding roughly 48 kB of pretty-printed JSON per run while discarding the one metric that could prove or refute the guard's premise. It is now attached only to frames that retried or degraded, so absence means a clean first attempt and presence is actionable.
- Regression strategy: the reworked contracts pin each previously-missed frame shape, the exhaustion path returning `degraded` instead of throwing, and that a clean first attempt neither settles nor annotates. A gapped-column fixture pins the run-length metric as the longest unbroken run rather than a per-column total — a reviewer mutation-proved that deleting the `columnRuns[x] = 0` reset still passed the original suite, because it only ever inspected 100%-black and 0%-black frames.
- Verification: `npm test` passes 47 files, and `npm run typecheck`, `npm run lint`, and `npm run build` are clean with only the established Phaser advisory. A real bounded 8-step `playtest:llm:visual-loop` against localhost wrote 9 of 9 screenshots with zero console errors, zero page errors, zero degraded findings, and zero frames carrying capture metadata, confirming the normal path stays clean and unannotated.
- Module split: adding the finding pushed `improvement-report.mjs` from 499 to 536 lines against the repo's 500-line ceiling, so its pure identity, evidence-shaping, action-tally, and JSON-hygiene helpers moved to `report-support.mjs`. The report is now 459 lines and the new module is listed in `VISUAL_LOOP_MODULES` so its source stays under the same review contracts.

## Scope boundary

- This commit changes only how the existing suite is scheduled. It does not touch game code, harness behavior, or any test's assertions, and it makes no claim about the in-flight visual-loop screenshot-stability work, which stays uncommitted and is being closed out separately.
- The green run reported here is the honest state of the declared gate on this machine. It is not a claim that the suite is fast, that its browser-heavy design is ideal, or that 16 concurrent Chromium suites is the right long-term shape for the suite.

## Broadened discovery, and closed the gap it found

- Why this lane: the recursive pass had reported `no-fix-candidate` with `broaden-discovery` as its required next action across several runs. The mechanism behind that verdict turned out to be structural rather than a sign of saturation — `coverageLedger` builds its `offered` set exclusively from `step.observation.availableActions`, so it detects seen-but-not-touched controls and can never report a state the heuristic cannot reach. Combined with the hard `MAX_VISUAL_LOOP_STEPS = 160` clamp, the declared scope cannot grow out of quiet on its own. `loop-ops/DIRECTIVES.md` has farm active with LLM exploration locked, so broadening had to come from a deterministic lane.
- Finding: the autosave/reload round trip was never exercised in a browser by anything in the repo. The visual loop installs `localStorage.clear()` as a context init script, which re-runs on every navigation, so `loadSavedFarmState()` always returns null and every pass boots a fresh farm — the loop exercises the write half of persistence and never the read half. `tests/persistence/localSave.test.ts` covers loading well, but only through an in-memory `MemoryStorage` double with no browser, Phaser, or DOM. Seven browser tests do call `page.reload()`, and every one asserts a DOM preference or a Farm Note; none asserted that farm simulation state survives.
- The lane had regressed rather than never existing: `scripts/llm-playtest/evaluate.mjs` still contains a live P1 `autosave-reload-lost-state` evaluator that diffs a `worker-care` scenario against a `post-reload` scenario. Nothing has produced those scenarios since `scripts/llm-playtest.mjs` became a shim that only re-imports the visual loop. The detector survived the migration as orphaned code, kept reachable solely by a unit test feeding it a synthetic literal.
- Underlying defect: `loadSavedFarmState()` collapsed three distinct outcomes into null — no save, JSON parse failure, and `isFarmState` rejection — and `main.ts` turned null into a brand-new farm through the same path as a first run, with no player-facing signal. Because `persistFarm()` runs on a ~1s autosave timer, the fresh farm then overwrote the rejected save about a second after boot, so the data was destroyed rather than merely skipped. Any schema change that made the game reject a save it wrote itself would silently delete a returning player's farm, and every gate would stay green.
- Fix: `loadFarmSave()` now reports `loaded`, `empty`, or `unreadable`; `loadSavedFarmState()` remains as a thin wrapper so the existing 30-plus unit assertions are untouched. Only `empty` — including storage that cannot be read at all — starts a new farm silently. For `unreadable`, the farm declines to destroy what it cannot repair: `autosaveBlocked` latches at startup, `persistFarm()` returns before writing, and the stored bytes are left untouched. The standing HUD warning needs no stickiness mechanism, because the early return is what keeps it on screen. `resetFarm()` clears the save and unlatches the block, making Reset the player's consent to lose it.
- Rejected design, and why: the first implementation copied the unreadable save to `farm.autosave.v1.unreadable` before starting fresh. Two independent adversarial reviewers refused it and were right on the code. A copy needs roughly twice the storage while the autosave that follows only replaces a key and needs none, so there is a real band where the copy throws `QuotaExceededError` and the overwrite then succeeds — the guard failed precisely on the large saves worth protecting. A reviewer measured the payload: history is about 97% of a save (`FARM_HISTORY_LIMIT` full core snapshots per direction), 139,858 B for a starter farm with 31 undo entries, projecting to ~9 MB at 400 tiles against a 5 MB origin quota, so that band is reachable in ordinary play. A second reviewer executed the repeat case and confirmed the second copy silently clobbers the first, discarding a `PRECIOUS-ORIGINAL` payload in favour of a throwaway farm while still claiming the old save was kept. Nothing in the product ever reads the copy back, so "kept for recovery" promised a path that does not exist, and `clearFarmSave()` never removed it, leaving an unbounded blob competing for the same quota. Declining to overwrite has none of these failure modes and is strictly less code.
- Regression anchors: `tests/browser/savePersistenceReload.test.mjs` proves a played farm restores across a real `page.reload()`, that an unreadable save is never overwritten, that Reset drops it and lets autosave resume, and that a genuine first run raises no false warning and still autosaves. A `sessionStorage` guard clears storage only on first boot, which is what makes a reload observable at all. Two mutation proofs, because both contracts pass on unfixed code for different reasons. Forcing `isFarmState` false — the exact schema-drift scenario — fails the reload contract with `expected 3 to be greater than or equal to 603`, a farm with 603 ticks restarting at tick 3. Removing the `autosaveBlocked` early return fails the no-overwrite contract with `expected '{"version":1,"tick":9,...}' to be '{"version":2,"tick":4242,"coins":999}'`, which is the data loss itself in the assertion output. The no-overwrite contract earns its name only under this design: the rejected copy-aside design wrote a different key from autosave, so its equivalent test passed with autosave entirely disabled and proved nothing. The reload contract identifies the farm by tick, tier, owned tiles, and lifetime watered/planted, so a partial restore that carries the tick across while dropping progress cannot read as success.
- Presentation defect caught by the required browser evidence, then generalized: the first draft of the notice ran 84 characters, and `.hud-alert` ellipsizes rather than wraps. At 1024x720 it measured `scrollWidth` 626 against `clientWidth` 486 and truncated, cutting exactly the half naming the way out. The alert's `clientWidth` there is ~486px against ~8.8px/char, so roughly 55 characters fit. The same defect already existed in two pre-existing messages: the autosave-failure warning ran 101 characters and truncated to "Autosave unavailable - progress remains in this...", losing the actionable instruction to free browser storage, and the reset-failure warning ran 85. All three are now at or under 53 characters. `tests/browser/persistenceFailure.test.mjs` gained a `test.each` over 1280x800 and 1024x720 that forces both failure warnings through the existing `Storage.prototype` stubs and measures rendered width; it reproduces red at 1024x720 against the old strings. That contract lives beside the stubbing machinery rather than in the save-reload file so it stays independent of the persistence design.
- Why the truncation survived this long: the existing failure assertions read `#hud` `textContent`, which holds the complete string no matter how it renders, so CSS ellipsis is invisible to them. Only comparing `scrollWidth` against `clientWidth` observes it.
- Verification: `npm test` passes 48 of 48 files with `npm run typecheck` and `npm run lint` clean and `npm run build` carrying only the established Phaser advisory. Headless evidence at both supported desktop sizes shows each full sentence reading in the HUD over a live farm.

## Scope boundary for the persistence lane

- This closes one deterministic discovery scope. It does not claim the loop's broader quality is evaluated, and it does not run a recursive pass — the fix is anchored by repo contracts rather than by a rerun comparison, because the loop structurally cannot reach the restore path it covers.
- A known adjacent gap stays open and is deliberately not claimed here: browser-level load validation and migration still only ever run against current-schema saves. A legacy-but-valid save — pre-ecology, missing `wildlife` — migrates in unit tests but has never been booted through Phaser, and a migrated state can be unit-valid and still throw in the renderer.
- The refusal to autosave is a deliberate product trade-off, accepted by the owner: a player whose save cannot be read gets no persistence at all until they Reset. The alternative was overwriting data they never agreed to lose. Because the block holds for the whole session, the standing warning occupies the HUD's only alert slot and suppresses tier and milestone guidance for that session — accepted here because autosave really is off and that is the most important fact on screen, and because the message names the exit.
- `speedPreference.test.mjs` "Space activates a focused button without also pausing the farm" flaked once under full-suite load with `expected 'Inventory' to match /Tier/`, then passed in isolation twice and in repeated full runs. It is an assertion race rather than a timeout, so it is a different class from the fork-cap fix above and is not claimed as addressed.
