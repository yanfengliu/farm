# 2026-07-15

## Restored the authoritative test gate

- Result: `npm test` was red on a clean checkout of `main` and had been reporting failures that no code change caused. On this 32-core machine the suite finished 14 files failed / 380 of 394 tests passing at commit `a0b6a6b`, with every failure an identical `Test timed out in 15000ms` rather than an assertion. The repo's first and most authoritative gate could not be trusted to distinguish a real regression from runner noise.
- Diagnosis: 16 suites under `tests/browser` each launch their own Chromium and their own Vite dev server, and the browser files set their own 15s per-test budgets inline (`}, 15000)`). The repo had no Vitest configuration of any kind, so Vitest defaulted to one fork per core and started ~31 of those suites at once. The tests were not wrong and the timeouts were not flakes in the tests themselves: the runner was oversubscribing the machine until browser work missed the budgets the tests declared for themselves.
- Fix: `vitest.config.ts` now caps `poolOptions.forks.maxForks` at `Math.max(2, Math.min(6, os.cpus().length - 1))`, so fork concurrency is bounded by the browser cost of the suite rather than by core count. Nothing else changed; no test timeout was raised and no test was skipped or weakened.
- Verification: the same suite at the same commit goes from 14 files failed to 46 of 46 files passing with the cap in place, and repeated full runs reproduce the green result rather than passing once by luck. Wall-clock cost is roughly 45-49s against ~40s for the previously-failing unbounded run, so the cap trades no meaningful speed for a trustworthy gate. `npm run typecheck` and `npm run lint` are clean and `npm run build` retains only the established Phaser chunk-size advisory.
- Regression anchor: `tests/browser/testRunnerConcurrency.test.mjs` pins that the fork cap stays a small bounded number and stays below the number of suites that launch their own Chromium, so the cap cannot silently regrow back to core count. A mutation that restores the uncapped `os.cpus().length - 1` fails the anchor with `expected 31 to be less than or equal to 8` and `expected 31 to be less than 17`; restoring the cap makes it green again.

## Scope boundary

- This commit changes only how the existing suite is scheduled. It does not touch game code, harness behavior, or any test's assertions, and it makes no claim about the in-flight visual-loop screenshot-stability work, which stays uncommitted and is being closed out separately.
- The green run reported here is the honest state of the declared gate on this machine. It is not a claim that the suite is fast, that its browser-heavy design is ideal, or that 16 concurrent Chromium suites is the right long-term shape for the suite.
