# 2026-07-16

## Decided the paused-command UX: edits apply immediately

- The canvas-placement work discovered that `submitFarmCommand` queues into the civ-engine world and commands apply only during a `step()`, while `main.ts` steps only when unpaused — so a paused farm silently deferred every player command with no UI signal, then burst all queued clicks on resume. Verified precisely before deciding: headlessly, three queued placements all apply in the first step after submission (deferred, not dropped, and bursty — one step drains the whole queue), and undo defers identically.
- Decision (option a): a paused farm applies player edits immediately by stepping exactly one recorded tick per command, implemented at the single UI-bridge choke point in `main.ts` — every surface (toolbar, canvas tools, hotkeys, panels) submits through it. The engine offers no public command-only flush (`processCommands` is protected), so one tick per edit is the boundary-respecting implementation: the tick is recorded normally, replay/history/reload-oracle all see ordinary ticks. The trade-off is honest and documented in the design doc: each deliberate paused edit advances 100ms of game time; the pause contract's spirit — the farm does not run on its own while you look — is preserved, and paused-actor-stability tests observe no commands so they are unaffected. Rejected: (b) queued-edit feedback UI explains a deferral that no longer needs to exist; (c) documentation alone leaves a trap that fooled this repo's own scripted player.
- `tests/browser/pausedCommands.test.mjs` (red-first, green 3x, mutation-proved by removing the paused-step guard) pins: a paused seed purchase lands while still paused, exactly one tick per edit, no free-running afterward, and undo of a paint while paused restores the cell. Writing it surfaced a design fact worth pinning in prose: `sellCrop` and `buySeeds` are deliberately outside the undo domain ("farm edits, upgrade purchases, and tier claims"), and the test's first draft wrongly asserted undo of a purchase — the probe showed `undoHistory=0` after buying, by design, not by bug.

## Gave the loop the save-reload-restore discovery scope

- The blind-spot analysis's core recommendation, built per the fleet build recipe (`building-recursive-loop.md` read first). `scripts/llm-visual-loop/reload-check.mjs` is the oracle: after every other evidence stage, read the autosave the game itself wrote, reload the page like a returning player, and compare farm identities. The baseline is deliberately the stored save rather than live state — what the save says is exactly what a returning player gets. Identity rules (tier, owned tiles, workers) hold exactly; monotonic rules (tick, coins, lifetime progress) may only move forward because the restored farm keeps simulating. A game-declared refusal (the unreadable-save HUD warning) is its own `save-refused` violation. Violations raise a critical `autosave-reload-lost-state` finding with the matching `data.class` — the stable bug-class id the fleet's prove-fixed granularity keys on — resurrecting the P1 oracle orphaned in `scripts/llm-playtest/evaluate.mjs` since that harness became a shim. No autosave present records an honest `skipped`.
- Ordering is structural: the check must run after bundle export and replay self-check because reloading destroys the in-page SessionRecorder. The contract test pins the source-order chain exportBundle → selfCheck → runSaveReloadCheck → evaluateVisualLoop.
- Building the negative proof caught a bug in the lane itself before it could ship a false positive: the loop's `addInitScript(localStorage.clear())` reruns on every navigation, so the check's own reload wiped the autosave and a *clean* build read as a disaster. The first seeded-bug drill "passed" for exactly the wrong reason — the finding fired from the wipe, not from the seeded rejection, visible in the packet because the post-reload alert showed milestone text instead of the unreadable warning. The clear is now guarded by a sessionStorage marker to once per context (the same idiom `speedPreference.test.mjs` uses), and the contract test pins the guard, since an unguarded clear silently reopens the exact blind spot this lane exists to close.
- Acceptance drill, both halves against real runs: seeded (`isFarmState` forced false) — an 8-step run raises critical `autosave-reload-lost-state` with violations `save-refused, tick-regressed, progress-regressed` and the alert reading "Saved farm unreadable - autosave off until Reset."; clean (mutation reverted) — saved tick 41 restores to tick 45 with zero violations and no finding. The fix and promotion halves of the class already exist as yesterday's persistence commit and its `savePersistenceReload` contracts.
- The pass declares the lane: `playtest-recursive.mjs`'s discovery scope id gains `-plus-save-reload` and its finding source names `save-reload-restore`, so a quiet pass's manifest records that this lane ran rather than letting the old scope string imply it.
- Full-pass proof: `npm run playtest:recursive` completed 149 of 160 decisions with a clean agent stop, zero console/page errors, zero findings, and a strong replay self-check (1 checked segment, 0 skipped, 0 divergences). The reload oracle checked a genuinely played farm — saved tick 5551 restored to 5567 — with zero violations, and the pass manifest records `local-heuristic-full-surface-160-step-plus-save-reload` with `no-fix-candidate`, `broaderGoalStatus: not-evaluated`, `nextAction: broaden-discovery`.
- `tests/browser/llmVisualLoopReloadCheck.test.mjs` (10 tests, red-first) pins identity extraction against a real snapshot, the comparison rules including the numbers carried into the finding packet, and the orchestration via a fake page: skip on no autosave, unparseable-save violation, clean restore, fresh-farm loss, game-refused save, and reload failure surfacing as `error` status instead of a throw. The module joins `VISUAL_LOOP_MODULES`, so the reviewability, line-ceiling, and no-debug-API contracts cover it; its first run tripped the pre-existing no-fixed-cap contract (`.slice(0, 60)` is a banned literal anywhere in the loop source) and the preview bound moved to 64.

## Deflaked the two intermittent browser suites

- Result: `speedPreference` "Space activates a focused button without also pausing the farm" and `annotations` "pins follow camera movement and View restores the captured composition" both failed intermittently — under full-suite load and, for the annotations test, occasionally in isolation. Both are now green five consecutive runs of the failing pair and two consecutive full-suite runs. No app code changed for either: both were test races against the same architectural fact, that every panel and HUD DOM update rides the next Phaser `ui.render()` frame.
- speedPreference was a genuine test-logic bug, not a tight budget. After pressing Space on the Goals tab it waited on the bare selector `#panel-content h2` — which is already satisfied by the still-rendered previous panel — then read the heading before the frame swapped Goals in, producing the recorded `expected 'Inventory' to match /Tier/`. The wait is now on the Goals heading text itself, and the speed read moved after that wait so a wrongly-paused farm would actually be visible to the assertion.
- The annotations failure took three instrumented probes to pin down, and the first two theories were wrong. It was not frame-stall budget exhaustion (raising the 2s waits to 5s made the failure *reliable* rather than fixing it, by letting the test reach the real blocker), and it was not per-frame panel churn (a MutationObserver recorded zero mutations across three seconds of an open draft). The trace that settled it: after saving note #1, the stale draft card is legally still in the panel for one frame because `saveDraft` refreshes the overlay synchronously while the panel rebuild waits for the next `ui.render()`. Around the second draft's open, the panel therefore transitions stale-save-card → aim-card → draft-card, and each rebuild detaches the Cancel node, so `page.click('[data-command="cancel-annotation"]')` can keep resolving nodes that are replaced beneath it — Playwright's `element was detached from the DOM, retrying` loop. A real player only ever sees the settled states.
- Fix: the cleanup clicks moved off the panel DOM entirely. `Escape` is the documented player path — the first returns the open draft to aiming, the second exits aiming — and the keyboard state machine is synchronous, so it cannot race a rebuild. The draft opening remains the actual assertion (the existing pin did not intercept the aim click), and the aria-pressed poll still verifies the exit state. The three tight 2000ms waits stay at 5000ms, sized for parallel-suite frame stalls; the test budget rose 30s → 45s.
- Evidence trail: `render_game_to_text()` state tracing showed the exact sequence healthy and stable for 2+ seconds (`aiming=false draft=true`, Cancel present and visible, textarea focused), which exonerated the app state machine and isolated the failure to click-time node identity.

## Closed the canvas tool-placement integration gap

- Gap (blind-spot report #4): the visual loop selects the Land, Well, Storage, and Bulldoze tools but never applies them through the canvas, and the headless suite drives `buyLand`/`placeBuilding` only as direct commands — so no test proved that a real click at real screen coordinates reaches the intended cell through the camera and changes both the simulation and the HUD.
- `tests/browser/canvasToolPlacement.test.mjs` closes it end to end. It calibrates screen-to-grid empirically: with Inspect selected, a click prints `Tile x, y` in the panel, so two separated clicks yield pixels-per-tile and an iterative correction loop verifies each target cell with Inspect before the real tool ever clicks it — a placement failure therefore cannot be excused by a mapping miss. It then places a well, places a storage bin and waits for the HUD capacity chip to follow, buys a locked adjacent tile and waits for the HUD coin chip to follow, and bulldozes the well back to empty, asserting simulation state at every step with zero page errors.
- The test deliberately does not pause the farm, and the reason is the session's main discovery: `submitFarmCommand` queues into the civ-engine world and commands only apply during a `step()`, while `main.ts` steps only when unpaused. A paused farm therefore silently defers every player command — placement, selling, undo — until resume, with no UI indication. The first draft of this test paused "for determinism" and every placement timed out; the placement machinery was fine all along. Diagnosis burned six probes (tool selection, overlay occlusion, guide-card interception, stale-DOM reads, cell mapping) before a headless reproduction isolated the queue. The test now plays unpaused like a real player and instead keeps targets away from the worker's position, path, and task target with a one-cell margin. The paused-commands behavior itself is flagged as a separate product decision (spawned follow-up task), not silently "fixed" here.
- Three test-observability traps from this file's own debugging are documented in comments: the Inspect panel keeps the previous inspection on screen (wait for text change, tolerate same-cell timeouts), Phaser processes pointer input on its next game step (wait on the state change, not after the click), and toolbar/HUD reads immediately after a click are one frame stale.
- Mutation proof: disconnecting only the Land case in `commandForTool` (returning null) fails the contract at the land step after the well and storage placements have already passed — specific, not a blanket failure. Restoring returns it to green across three consecutive runs.

## Opened the legacy-save browser boot lane

- Gap (named in yesterday's summary): every browser test that seeds a save uses a current-schema snapshot, so `normalizeFarmState`'s additive migrations had never run in a browser — only against the unit suite's in-memory storage double. A migrated state can be unit-valid and still throw in the renderer, and nothing would have caught that class.
- `tests/browser/legacySaveMigration.test.mjs` seeds a genuinely old save built by stripping every validation-optional field a pre-Village-Harvest, pre-ecology farm lacks — `wildlife`, `community`, `upgrades`, both optional lifetime counters, and every `pumpkin` crop-record entry, including inside a stripped undo snapshot — then boots it through the real path: `main.ts`, Phaser, autosave. It asserts the save loads rather than being refused as unreadable, the played tick survives, every migration lands (Pip and Mallow with four fish, zeroed upgrades, community state, pumpkin entries), the migrated ecology reaches `render_game_to_text()`, the canvas actually paints, zero page errors, and — closing the round trip — autosave writes back a current-schema save so the next boot is no longer a legacy boot.
- Mutation proof: commenting out `state.wildlife ??= createInitialWildlifeState()` in `normalizeFarmState` fails the contract with `TypeError: Cannot read properties of undefined (reading 'ducks')` — precisely the unit-valid-but-breaks-downstream failure class the lane exists to catch. Restoring the line returns it to green.
